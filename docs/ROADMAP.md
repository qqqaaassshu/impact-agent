# 后续规划

## 1. 当前版本定位

当前版本定位为培训第一阶段可交付的智能体最小可用版本。

当前重点：

- 聚焦字段变更 `field_rename`
- 面向大型前端项目
- 以本地代码源为主要输入
- 以确定性扫描作为主路径
- 以有限 LLM 复核作为辅助能力
- 通过 Web 页面展示分析阶段和评估结果

当前版本不定位为生产级平台。

## 2. 提交前完善事项

提交前建议重点完善评审材料和演示链路：

- 准备稳定演示用例，例如 `amount` 变更为 `totalAmount`
- 准备 Web 分析截图，包含阶段进度、证据链、风险等级和置信度
- 补充正式 MCP Tool 封装方案或最小实现
- 保持 README 中已实现能力和后续规划的边界区分
- 保留后端单测和前端构建结果

## 3. 第二阶段能力增强

若进入进阶集训，建议围绕推理可靠性和业务规则注入进行增强。

重点方向：

- 增加反思节点，对高风险结论进行二次校验
- 注入项目级业务规则，例如字段命名规范、接口字段规范、路径规范
- 增强不确定项解释能力
- 支持人工反馈，将误报和漏报沉淀为项目级规则
- 保存 LLM 判断输入、输出和原因，便于审计和复盘

## 4. 扫描能力规划

第一阶段：当前能力

- `ripgrep` 多关键词搜索
- 文件类型过滤
- 注释命中排除
- Vue / React 上下文识别
- 简单变量传递候选识别
- 已召回文件的轻量 AST 结构分析

第二阶段：索引缓存

- 记录文件指纹
- 缓存字段命中索引
- 缓存组件路径和接口路径
- 重复扫描仅处理变更文件

第三阶段：AST 增强

- 更完整的 TypeScript AST 字段引用识别
- import alias 解析
- interface / type / schema 关系追踪
- 对象解构和字段重命名识别

第四阶段：关系图分析

- 页面到组件关系
- 组件到 API 关系
- 表格列、表单项、schema 到字段关系
- handler 到接口调用关系

## 5. 变更类型规划

`field_rename`：

- 当前主链路
- 后续持续提升准确率、召回率和性能

`feature_change`：

- 后续扩展方向
- 需要定义更严格的输入格式
- 需要识别页面入口、事件、handler、API、权限点和刷新链路

`api_contract_change`：

- 可作为字段变更的扩展方向
- 关注接口返回结构、请求参数、类型定义和 mock

`permission_change`：

- 可作为功能变更子类
- 关注权限码、按钮显隐、路由守卫和接口鉴权

## 6. MCP 规划

当前项目已提供前端检索 Skill 和本地扫描能力。后续可将扫描能力封装为正式 MCP Tool。

建议 Tool 定义：

- Tool 名称：`frontend_impact_search`
- 输入：`repo_root`、`repo_path`、`keywords`、`file_extensions`、`max_results`
- 输出：归一化命中列表，包含文件、行号、上下文、命中词和命中类型
- 约束：只读、不修改仓库、不调用 LLM、不生成最终结论

Agent 主流程继续负责：

- 判断是否调用 Tool
- 解释 Tool 返回的证据
- 判断是否进入 LLM 复核
- 生成最终影响范围报告

## 7. Web 产品规划

短期能力：

- 展示分析阶段和耗时
- 展示 LLM 参与阶段
- 展示注释命中和排除原因
- 展示高风险上下文复核结果

中期能力：

- 支持历史详情页
- 支持报告导出
- 支持配置扫描子路径
- 支持调整 LLM 复核上限
- 支持人工标记误报和漏报

长期能力：

- 接入统一登录
- 接入项目列表
- 接入 MR / CI 流程
- 接入团队级规则库

## 8. 生产化规划

生产化阶段需补齐以下能力：

- 权限控制
- 审计日志
- 超时和取消机制
- 资源使用限制
- 任务队列
- 扫描缓存
- 敏感信息脱敏
- 模型调用记录
- 报告归档

生产环境中，LLM 访问代码和业务数据前应完成公司合规评估，并结合实际要求进行脱敏和审计。

## 9. 暂缓事项

短期暂缓以下事项：

- 同时扩展多个变更类型
- 让 LLM 直接读取完整仓库
- 自动修改代码并提交
- 引入复杂多 Agent 协作
- 同步接入多个远程代码平台
- 在无证据支持的情况下生成影响结论

当前优先目标是提升字段变更影响分析的准确率、性能和可复核性。
