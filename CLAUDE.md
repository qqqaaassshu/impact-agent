# impact-agent

本文档是 Claude Code CLI 在本仓库中的项目级工作说明。开发、调试、评审和文档更新应优先遵循本文档约定。

## 项目定位

这是一个面向前端代码仓库的“需求变更影响范围评估 Agent”。

当前阶段的产品定义必须收敛为：

- 只完整支持字段变更 `field_rename`
- 功能变更 `feature_change` 只保留扩展接口与设计方向，不承诺当前可用
- 优先支持大型前端代码仓库中的只读影响评估

系统目标不是开放式聊天，也不是泛化代码问答，而是：

- 接收一个明确的工程路径和需求描述
- 判断该需求是否属于当前支持范围
- 对支持的需求执行前端代码检索与影响分析
- 输出结构化、可追溯、可复核的评估报告

## 当前交付范围

当前项目定位为培训第一阶段可交付的智能体最小可用版本，尚不属于生产级平台。

能力边界说明：

- 已实现：字段变更识别、本地仓库扫描、Vue/React 证据分类、有限 LLM 复核、Web 流式展示、结构化报告
- 已具备：本地前端搜索 Skill、确定性扫描引擎、已召回文件 AST 分析、LangGraph 编排、风险与置信度策略
- 未完成：正式 MCP Tool 服务、GitLab 代码源、全量 AST 深度分析、自动修复、自动 MR、生产权限体系

如果评审要求 MCP 形态，应优先将已有本地检索能力封装为 MCP Tool，避免重新实现既有分析链路。

## 当前阶段范围

当前版本必须明确支持：

- 变更类型：`field_rename`
- 代码源：本地代码源 `local`
- 前端框架：Vue / React
- 文件类型：`.ts` `.tsx` `.js` `.jsx` `.vue` `.json`

当前版本暂不承诺完整支持：

- `feature_change`
- GitLab 代码源
- 全量 AST 深度分析
- 自动修复、自动提交、自动开 MR、自动部署

如果需求不属于当前支持范围，必须显式拒绝，并说明原因。

## 核心运行流程

主流程必须围绕下面这条链路构建：

1. 用户输入工程路径与需求描述
2. `intake` 将输入规范化为结构化请求
3. 判断当前是否支持该需求
4. 若支持，则触发前端代码检索流程
5. 基于检索结果做字段变更影响分类
6. 对不确定项显式输出，并生成证据链
7. 输出最终结构化 JSON 报告

当前版本中，用户视角下的典型流程应当是：

- 输入工程路径
- 输入字段变更需求
- Agent 判断是否属于 `field_rename`
- Agent 调用前端搜索 Skill / 扫描引擎执行代码检索
- Agent 对候选结果做影响分类
- Agent 输出 `confirmed_affected / excluded / uncertain / evidence_chain / risk_level / overall_confidence`

## Skill、扫描引擎与 LLM 边界

必须明确区分：

- `Skill`：定义前端代码检索工作流、输入输出规范、调用顺序、检索注意事项
- `扫描引擎 / 搜索脚本`：真正执行本地代码检索、文件过滤、关键词搜索、结果归一化
- `Agent 主流程`：决定何时调用 Skill、如何解释结果、如何输出报告
- `LLM`：用于有限范围的辅助判断，不作为主扫描器

Skill 不应被定义为仅依赖提示词完成全仓扫描的能力主体。

推荐做法是：

- 前端检索 Skill 负责沉淀流程和边界
- Skill 内部引用脚本、规则说明、检索模板
- 具体搜索能力由 Python 脚本或确定性代码模块实现
- LLM 只能复核已有 evidence，不能直接新增命中文件

## LLM 使用原则

LLM 可以参与：

- 自然语言需求转结构化请求
- 判断当前需求是否属于 `field_rename`
- 补充少量搜索线索
- 对动态引用、变量传递、模糊引用做局部语义仲裁
- 生成最终摘要说明

LLM 不应参与：

- 全仓库扫描
- 大量文件的主分类判断
- import / alias / 路径解析的主事实判断
- 代码事实的主来源判断

主结论必须以代码证据为准，不能以 LLM 猜测为准。

## 架构约束

开发时必须保持以下边界清晰：

- `orchestrator` 只负责流程控制和阶段调度，不写业务匹配细节
- `strategies` 定义不同变更类型的分析方式
- `adapters/code_source` 负责代码从哪里来，以及如何读取和搜索
- `adapters/framework` 负责 Vue / React 代码如何解释
- `policies` 负责风险和置信度规则
- `services/report_builder` 负责最终 JSON 报告结构
- `services/knowledge_store` 负责项目画像、索引缓存、历史记录、人工反馈
- `Skill` 负责组织前端检索工作流，不直接替代代码扫描模块

以下逻辑不得直接分散在主流程中：

- Vue 专属判断
- React 专属判断
- GitLab 专属判断
- 关键词词库
- 路径规则
- 风险阈值
- 检索流程细节

相关逻辑应分别归属到：

- framework analyzer
- code source adapter
- strategy
- policy
- skill / script / 配置

## 项目扫描原则

当前项目必须朝“大仓库可用”的方向设计扫描能力。

扫描能力必须遵守：

- 优先做范围收缩，而不是默认全仓深扫
- 优先做确定性检索，而不是先让 LLM 读代码
- 先召回候选，再做深分析
- 只对不确定项做 LLM 仲裁
- 所有 LLM 复核都必须有数量上限

扫描阶段至少应考虑：

- 工程路径和代码源快照
- 文件过滤
- 关键词搜索
- 候选结果归一化
- 模板 / JSX / 对象属性 / 类型 / mock / schema 等静态模式识别
- 已召回 JS / TS 文件的 AST 结构识别
- 注释命中识别与排除
- 同文件局部变量传递识别

后续若扩展索引、关系追踪、调用链分析，应作为增强能力沉淀到独立服务中，不应固化在 orchestrator 中。

## 变更类型规则

### field_rename

重点关注：

- 模板 / JSX 字段引用
- 对象属性访问
- 类型定义
- API 字段
- mock / fixture
- schema 配置
- 列配置 / 表单配置 / 映射配置
- 简单变量传递

分类时必须区分：

- `confirmed_affected`
- `excluded`
- `uncertain`

如果无法静态确认，必须进入 `uncertain`，不能强行输出确认结论。

### feature_change

当前版本不承诺主链路可用。

后续实现时重点关注：

- 页面入口
- 动作按钮或操作列
- 事件绑定
- handler
- API 调用
- 权限点
- 刷新链路
- 调用链

## 框架支持规则

Vue 和 React 都是一等支持对象。

不得假设：

- 页面只在 `src/views`
- 事件绑定只有 Vue 写法
- 所有列表操作都在同一个组件里
- 所有 API 都是固定目录固定命名

前端框架差异必须通过 `FrameworkAnalyzer` 处理，不应固化在主流程中。

## 代码源规则

当前主实现优先支持：

- 本地项目目录

后续扩展支持：

- GitLab project/ref

代码源差异必须通过 `CodeSourceAdapter` 处理，不应固化在 strategy 或 orchestrator 中。

如果分析的是本地代码：

- 要记录当前 commit
- 要记录是否存在未提交变更

如果分析的是 GitLab：

- 要记录 project_id
- 要记录 ref
- 尽量解析到固定 commit

## 输出规则

所有结论必须能回溯到证据。

当前必须输出：

- `summary`
- `coverage`
- `confirmed_affected`
- `excluded`
- `uncertain`
- `evidence_chain`
- `risk_level`
- `overall_confidence`

其中：

- `confirmed_affected` 表示有足够代码证据确认受影响
- `excluded` 表示有足够证据确认不属于本次影响范围
- `uncertain` 表示存在动态引用、信息不足、或静态分析无法确认

## Web 展示规则

Web 页面必须使用中文展示主要信息。

Web 端应展示：

- 当前分析阶段
- 是否进入 LLM 需求判断
- 是否进入 LLM 搜索线索补充
- 是否进入 LLM 特殊上下文复核
- 注释命中必须明确显示为注释并排除
- 不展示测试数据作为默认输入或默认历史

历史记录应优先只展示 Web 入口产生的记录，避免 CLI 测试数据污染用户演示。

## 开发规则

- 当前阶段限定为只读分析，不修改被分析仓库代码
- 不实现自动修复、自动提交、自动开 MR、自动部署
- 优先保证主流程、Skill 集成、扫描引擎、输出结构稳定，再逐步加精度
- 每条核心规则都应该可测试
- 优先采用小函数和可组合实现，避免形成职责过重的单一文件
- 不为未来假设场景做过度抽象
- 当前阶段优先实现可用于大型仓库的字段变更分析能力，不在当前阶段构建泛化分析能力

## 测试要求

当前阶段至少覆盖：

- Vue 字段变更
- React 字段变更
- 不确定项识别
- 证据链生成
- 本地代码源适配
- 需求支持性判断
- Skill 触发后的检索结果归一化
- 注释命中识别
- 变量传递候选识别
- LLM 复核上限
- Web 流式状态

后续扩展时再补：

- Vue 功能变更
- React 功能变更
- GitLab 代码源适配

## 当前实现优先级

按以下顺序推进：

1. request / report / state / evidence 模型
2. intake：判断需求是否属于 `field_rename`
3. 本地代码源适配器
4. 前端代码检索 Skill
5. 本地搜索脚本 / 扫描引擎
6. 字段变更策略
7. Vue analyzer
8. React analyzer
9. 风险与置信度 policy
10. report builder
11. knowledge store
12. Web 流式展示
13. MCP Tool 封装
14. feature_change 扩展设计
15. GitLab 代码源扩展
